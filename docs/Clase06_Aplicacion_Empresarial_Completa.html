DIAPOSITIVA 1 — Arquitectura integrada
🏗️ La arquitectura de 4 capas en Jakarta EE

Una aplicación empresarial Jakarta EE organiza su código en capas con responsabilidades bien definidas. Cada capa solo conoce a la inmediatamente inferior. Nunca se salta capas.

Validaciones en Jakarta EE
DIAPOSITIVA 5 — Los 3 niveles de validación
1️⃣ Vista (JSF)

Dónde: Archivo .xhtml

Qué valida:

  • required="true"
  • f:validateRegex
  • f:validateLength
  • Conversores de tipo

Se ejecuta en Fase 3 del ciclo JSF

2️⃣ Entidad (Bean Validation)

Dónde: Clase Java con anotaciones

Qué valida:

  • @NotBlank
  • @Email
  • @Size(max=100)
  • @NotNull

Integrado con JPA y JSF

3️⃣ Negocio (Servicio)

Dónde: Clase Servicio (@ApplicationScoped)

Qué valida:

  • Email único en BD
  • Documento único en BD
  • Reglas de negocio complejas
  • Consistencia de datos

Consulta la BD para validar

Bean Validation 3.0
DIAPOSITIVA 7 — Anotaciones de Bean Validation
✅ ¿Qué es Bean Validation?

Bean Validation es el estándar Jakarta EE para declarar restricciones sobre los campos de una clase Java usando anotaciones directamente en la entidad. Se integra automáticamente con JPA (valida antes del INSERT/UPDATE) y con JSF (valida en Fase 3).

AnotaciónAplica aDescripciónEjemplo
@NotNullCualquier tipoEl valor no puede ser null@NotNull LocalDateTime fechaHora
@NotBlankStringNo null, no vacío, no solo espacios@NotBlank String nombre
@NotEmptyString, CollectionNo null y no vacío@NotEmpty List<?> items
@SizeString, CollectionLongitud mínima y/o máxima@Size(min=2, max=100)
@EmailStringFormato de email válido@Email String email
@Min / @MaxNuméricoValor mínimo / máximo@Min(1) @Max(480) Integer duracion
@DecimalMinDecimalValor mínimo decimal@DecimalMin("0.00") BigDecimal precio
@PatternStringDebe cumplir expresión regular@Pattern(regexp="[MH]") String sexo
@Past / @FutureFecha/HoraDebe ser pasado / futuro@Past LocalDate fechaNac
DIAPOSITIVA 8 — Bean Validation en Cliente.java
📄 Cliente.java — Anotaciones de validación
@Entity
@Table(name = "cliente")
public class Cliente implements Serializable {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    // @NotBlank garantiza que no sea null, vacío ni solo espacios
    @NotBlank(message = "El documento es obligatorio")
    @Size(max = 20, message = "Máximo 20 caracteres")
    @Column(nullable = false, unique = true, length = 20)
    private String documento;

    @NotBlank(message = "El nombre es obligatorio")
    @Size(max = 100, message = "Máximo 100 caracteres")
    @Column(nullable = false, length = 100)
    private String nombre;

    // @Email valida el formato de correo electrónico
    @NotBlank(message = "El email es obligatorio")
    @Email(message = "El formato del email no es válido")
    @Size(max = 150)
    @Column(nullable = false, unique = true, length = 150)
    private String email;

    // @Column(nullable=false) + @NotBlank → doble protección
    // Bean Validation actúa ANTES que la BD (más eficiente)
}Cliente.java
DIAPOSITIVA 9 — Bean Validation en Turno.java
📄 Turno.java — Validaciones de fecha y relaciones
@Entity
@Table(name = "turno")
public class Turno implements Serializable {

    // @NotNull: fecha obligatoria (no aplica @NotBlank en LocalDateTime)
    @NotNull(message = "La fecha y hora son obligatorias")
    @Column(name = "fecha_hora", nullable = false)
    private LocalDateTime fechaHora;

    // @Size en String de texto libre
    @Size(max = 200, message = "El motivo no puede superar 200 caracteres")
    @Column(length = 200)
    private String motivo;

    // En relaciones: @ManyToOne con optional=false actúa como @NotNull
    // JPA valida que mascota no sea null antes de hacer persist
    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "mascota_id", nullable = false)
    private Mascota mascota;
}Turno.java
✅
Doble protección: Usar Bean Validation + restricciones de columna (nullable=false, unique=true) da doble capa de seguridad. Bean Validation actúa en Java (más rápido), y las restricciones SQL son la última barrera a nivel de base de datos.

Manejo de excepciones
DIAPOSITIVA 10 — Estrategia por capas
🛡️ ¿Dónde se maneja cada excepción?

En una arquitectura de capas, cada nivel tiene su responsabilidad ante los errores. El objetivo: que el usuario siempre reciba un mensaje claro, y que los detalles técnicos queden en los logs del servidor.

CapaTipo de excepciónAcción
Servicio IllegalArgumentException Lanzar con mensaje de negocio claro. Ej: "Email duplicado"
Servicio IllegalStateException Lanzar cuando una transición de estado no es válida. Ej: "Turno ya cancelado"
DAO PersistenceException Dejar que suba. El contenedor hace ROLLBACK automático
Bean IllegalArgumentException Capturar y mostrar como SEVERITY_WARN en la vista
Bean Exception (genérica) Capturar y mostrar como SEVERITY_ERROR con mensaje genérico
DIAPOSITIVA 11 — Patrón try-catch en el Bean
📄 ClienteBean.java — Manejo correcto
public void guardar() {
  try {
    // El servicio valida y persiste
    clienteServicio.guardar(clienteSeleccionado);
    cargarClientes();
    clienteSeleccionado = new Cliente();
    modoEdicion = false;
    addMsg(SEVERITY_INFO,
      "Éxito", "Cliente guardado.");

  } catch (IllegalArgumentException e) {
    // Error de negocio — mensaje claro al usuario
    addMsg(SEVERITY_WARN,
      "Atención", e.getMessage());

  } catch (Exception e) {
    // Error técnico — mensaje genérico
    addMsg(SEVERITY_ERROR,
      "Error",
      "No se pudo guardar. Intente nuevamente.");
  }
}
📄 ClienteServicio.java — Lanzar excepción
@Transactional
public void guardar(Cliente cliente) {

  // Validación de negocio → lanza excepción
  clienteDAO.buscarPorEmail(cliente.getEmail())
    .ifPresent(existente -> {
      boolean mismo = existente.getId()
        .equals(cliente.getId());
      if (!mismo)
        // @Transactional hace ROLLBACK automático
        // cuando se lanza RuntimeException
        throw new IllegalArgumentException(
          "Email duplicado: " + cliente.getEmail()
        );
    });

  if (cliente.getId() == null)
    clienteDAO.guardar(cliente);
  else
    clienteDAO.actualizar(cliente);
}
DIAPOSITIVA 12 — Errores frecuentes y soluciones
Error en runtimeCausa más comúnSolución
TransactionRequiredException Llamar a persist/merge/remove sin @Transactional Agregar @Transactional al método del DAO o Servicio
LazyInitializationException Acceder a colección LAZY fuera del contexto JPA Usar JOIN FETCH en el JPQL o cambiar a EAGER
ConstraintViolationException Violar una restricción UNIQUE o NOT NULL en MySQL Validar antes en el Servicio con una consulta previa
ELException: null Bean no inyectado o nombre incorrecto en EL Verificar @Named, beans.xml y nombre del bean
ConverterException El valor del selectOneMenu no se puede convertir al tipo esperado Agregar <f:converter converterId="jakarta.faces.Long"/>
ViewExpiredException La vista JSF expiró (recarga después de POST) Usar <h:link> en lugar de <a href> para navegar
⚠️
Nunca mostrar el stack trace al usuario. El catch genérico (catch(Exception e)) muestra un mensaje amigable. Los detalles técnicos deben ir al log del servidor: Logger.getLogger(getClass().getName()).log(Level.SEVERE, "Error", e)

Buenas prácticas
DIAPOSITIVA 13 — Principios SOLID aplicados
PrincipioQué diceCómo lo aplicamos
S Single Responsibility Una clase, una responsabilidad ClienteBean solo maneja la vista. ClienteServicio solo el negocio. ClienteDAO solo la BD.
O Open/Closed Abierto para extensión, cerrado para modificación GenericDAO se extiende sin modificarse. Se agregan nuevos DAOs sin tocar los existentes.
L Liskov Substitution Las subclases deben poder reemplazar a sus padres ClienteDAO extends GenericDAO. Donde se use GenericDAO puede usarse ClienteDAO.
I Interface Segregation Interfaces específicas en lugar de generales Servicios separados por entidad. ClienteServicio no conoce MascotaDAO.
D Dependency Inversion Depender de abstracciones, no implementaciones @Inject (CDI) gestiona las dependencias. Bajo acoplamiento entre capas.

DIAPOSITIVA 15 — Scopes CDI: cuándo usar cada uno
ScopeDuraciónUso recomendadoSerializable
@RequestScoped Una sola petición HTTP Beans sin estado que solo procesan y redirigen Opcional
@ViewScoped Mientras el usuario está en la misma vista Formularios con tabla en la misma página (ABM) Sí, obligatorio
@SessionScoped Toda la sesión del usuario Datos de sesión: usuario logueado, carrito de compras Sí, obligatorio
@ApplicationScoped Toda la vida de la aplicación Servicios, DAOs, catálogos estáticos No necesario
@Dependent El del componente que lo inyecta Beans auxiliares sin estado propio No necesario
💡
Regla de oro para los scopes: Si el Bean tiene un formulario más una tabla en la misma página → @ViewScoped. Si guarda datos del usuario entre páginas → @SessionScoped. Si es un servicio sin estado → @ApplicationScoped.

Relaciones JPA: @OneToMany y @ManyToOne
DIAPOSITIVA 18 — Tipos de relaciones JPA

@ManyToOne Muchos registros → uno En la tabla del "muchos" (FK) Muchas Mascotas → un Cliente

DIAPOSITIVA 19 — @ManyToOne en Mascota
📄 Mascota.java — lado "muchos" (tiene la FK)
@Entity
@Table(name = "mascota")
public class Mascota implements Serializable {

    // @ManyToOne: muchas mascotas → un cliente
    // FetchType.LAZY: NO carga el cliente
    // hasta que se acceda a mascota.getCliente()
    @ManyToOne(
        fetch    = FetchType.LAZY,
        optional = false  // = NOT NULL
    )
    @JoinColumn(
        name     = "cliente_id", // nombre de la FK
        nullable = false
    )
    private Cliente cliente;
    // EclipseLink genera:
    // FOREIGN KEY (cliente_id) REFERENCES cliente(id)
}
📄 Cliente.java — lado "uno" (referencia inversa)
@Entity
@Table(name = "cliente")
public class Cliente implements Serializable {

    // @OneToMany: un cliente → muchas mascotas
    // mappedBy: el campo en Mascota que tiene la FK
    // cascade=ALL: si elimino cliente, elimina mascotas
    // orphanRemoval: elimina mascotas sin cliente
    @OneToMany(
        mappedBy      = "cliente",
        cascade       = CascadeType.ALL,
        fetch         = FetchType.LAZY,
        orphanRemoval = true
    )
    private List<Mascota> mascotas
        = new ArrayList<>();
    // NO genera columna extra en tabla cliente
    // La FK ya está en tabla mascota
}
DIAPOSITIVA 20 — FetchType: LAZY vs EAGER
✅ LAZY (recomendado) — Carga bajo demanda
@ManyToOne(fetch = FetchType.LAZY)
private Cliente cliente;

// JPA genera UNA sola consulta al buscar mascotas:
// SELECT * FROM mascota WHERE id = 1
//
// El cliente se carga SOLO si se llama:
// mascota.getCliente().getNombre()
// → en ese momento genera:
// SELECT * FROM cliente WHERE id = ?

Ventaja: Menos datos en memoria. Solo carga lo que se necesita.

⚠️ EAGER — Carga inmediata con JOIN
@ManyToOne(fetch = FetchType.EAGER)
private Cliente cliente;

// JPA genera JOIN automático al buscar mascotas:
// SELECT m.*, c.* FROM mascota m
// INNER JOIN cliente c ON m.cliente_id = c.id
// WHERE m.id = 1
//
// El cliente SIEMPRE se carga, lo uses o no

Cuidado: Si hay muchos niveles de relaciones EAGER puede cargar demasiados datos.

DIAPOSITIVA 21 — El problema N+1 y JOIN FETCH
❌ Problema N+1 — Sin JOIN FETCH
// Esta query trae 100 mascotas con LAZY cliente:
"SELECT m FROM Mascota m"
// → 1 SELECT para las mascotas
// → 100 SELECT más (uno por cada mascota para cargar su cliente)
// → 101 consultas en total = problema N+1 ❌
✅ Solución — JOIN FETCH en JPQL
// JOIN FETCH fuerza la carga en UNA sola consulta SQL:
"SELECT m FROM Mascota m JOIN FETCH m.cliente ORDER BY m.nombre"
// → genera: SELECT m.*, c.* FROM mascota m
//           INNER JOIN cliente c ON m.cliente_id = c.id
//           ORDER BY m.nombre
// → 1 sola consulta, sin importar cuántas mascotas haya ✅

// Para 3 niveles (Turno → Mascota → Cliente):
"SELECT t FROM Turno t "
+ "JOIN FETCH t.mascota m "
+ "JOIN FETCH m.cliente "
+ "ORDER BY t.fechaHora DESC"

📄 TurnoServicio.java — Validación de estados
@ApplicationScoped
public class TurnoServicio {

    @Inject private TurnoDAO   turnoDAO;
    @Inject private MascotaDAO mascotaDAO;

    /** Reserva un nuevo turno. Valida fecha futura y mascota activa */
    @Transactional
    public void reservar(Turno turno, Long mascotaId) {
        if (turno.getFechaHora().isBefore(LocalDateTime.now()))
            throw new IllegalArgumentException(
                "No se puede reservar un turno en el pasado.");
        Mascota mascota = mascotaDAO.buscarPorId(mascotaId);
        if (mascota == null || !mascota.isActivo())
            throw new IllegalArgumentException(
                "La mascota no existe o está inactiva.");
        turno.setMascota(mascota);
        turno.setEstado(Turno.PENDIENTE);
        turnoDAO.guardar(turno);
    }

    /** PENDIENTE → CONFIRMADO */
    @Transactional
    public void confirmar(Long id) {
        Turno t = obtenerYValidar(id);
        validarTransicion(t, Turno.PENDIENTE, Turno.CONFIRMADO);
        turnoDAO.cambiarEstado(id, Turno.CONFIRMADO);
    }

    /** CONFIRMADO → REALIZADO */
    @Transactional
    public void marcarRealizado(Long id) {
        Turno t = obtenerYValidar(id);
        validarTransicion(t, Turno.CONFIRMADO, Turno.REALIZADO);
        turnoDAO.cambiarEstado(id, Turno.REALIZADO);
    }

    private void validarTransicion(Turno t,
                                      String estadoOrigen,
                                      String estadoDestino) {
        if (!estadoOrigen.equals(t.getEstado()))
            throw new IllegalStateException(
                "El turno debe estar en " + estadoOrigen
                + " para pasar a " + estadoDestino);
    }
}TurnoServicio.java

DIAPOSITIVA 29 — Resumen: lo que aprendimos hoy
🏗️ Arquitectura integrada
  • 4 capas: Vista → Bean → Servicio → DAO
  • Cada capa tiene una sola responsabilidad
  • El Bean nunca habla directamente al DAO
  • CDI gestiona todas las dependencias con @Inject
✅ Validaciones
  • 3 niveles: Vista → Entidad → Negocio
  • @NotBlank, @Email, @Size en la entidad
  • required, f:validateRegex en la vista
  • Unicidad (email/doc) validada en el Servicio
🛡️ Excepciones y estado
  • IllegalArgumentException para errores de negocio
  • try/catch en el Bean con mensaje al usuario
  • @Transactional hace ROLLBACK en RuntimeException
  • Nunca mostrar stack trace al usuario final
🔗 Relaciones JPA
  • @ManyToOne en el lado "muchos" (tiene la FK)
  • @OneToMany con mappedBy en el lado "uno"
  • FetchType.LAZY por defecto (performance)
  • JOIN FETCH en JPQL para evitar N+1
