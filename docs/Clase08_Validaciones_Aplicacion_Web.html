¿Por qué validar?
DIAPOSITIVA 1 — El costo de no validar
💥 Sin validación — Problemas de integridad
  • Email vacío → no se puede contactar al cliente
  • Documento duplicado → registros inconsistentes
  • Fecha de turno en el pasado → confusión en la agenda
  • Precio negativo → errores contables
  • Texto de 5000 caracteres en campo de 100 → DataTruncation
⚠️ Sin validación — Problemas de seguridad
  • SQL Injection: '; DROP TABLE cliente; --
  • XSS: <script>alert('hack')</script>
  • Subir archivos ejecutables disfrazados de imágenes
  • Acceder a IDs de otros usuarios sin autorización
  • Crear datos inválidos que rompen la lógica de negocio
✅ Con validación — Beneficios
  • Datos limpios y consistentes en la BD
  • Mensajes de error claros al usuario
  • Protección contra ataques comunes
  • Aplicación más robusta y confiable
  • Menos errores en producción
DIAPOSITIVA 2 — Principio de defensa en profundidad
🔐
Defensa en profundidad: Ninguna capa es suficiente por sí sola. Si la validación de la vista falla (el usuario desactiva JavaScript o usa una herramienta como Postman), la validación del servidor debe detener el dato inválido. Si el servidor falla, la base de datos es la última barrera.

DIAPOSITIVA 4 — Ejemplo real: Alta de un Cliente
🌐 Vista (JSF) — Nivel 1
El campo "Email" tiene required="true" y f:validateRegex pattern="^[^@]+@[^@]+\.[^@]+$".
Si el usuario deja el campo vacío o escribe "nombre_sin_arroba" → JSF muestra el error en la vista sin ir al servidor.
✅ Entidad Cliente.java — Nivel 2
El campo email tiene @NotBlank y @Email.
Si alguien llama al servicio directamente sin pasar por JSF → Bean Validation lanza ConstraintViolationException antes de llegar a la BD.
⚙️ ClienteServicio.java — Nivel 3
El servicio consulta: clienteDAO.buscarPorEmail(email).
Si ya existe un cliente con ese email → lanza IllegalArgumentException("Ya existe un cliente con ese email").
Esta validación no puede hacerla Bean Validation porque requiere consultar la base de datos.
🔌 JPA/EclipseLink — Nivel 4
La columna está mapeada con @Column(unique=true, nullable=false).
Si falla alguna validación anterior y aún así se intenta persistir → JPA lanza PersistenceException.
🗄️ MySQL — Nivel 5
La columna email en MySQL tiene NOT NULL y UNIQUE.
Si se intenta insertar directamente con SQL sin pasar por la aplicación → MySQL rechaza el dato. Es la última barrera absoluta.

DIAPOSITIVA 8 — Bean Validation en las entidades del sistema
📄 Cliente.java — Bean Validation completo
@Entity @Table(name = "cliente")
public class Cliente implements Serializable {

    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @NotBlank(message = "El documento es obligatorio")
    @Size(max = 20, message = "El documento no puede superar 20 caracteres")
    @Column(nullable = false, unique = true, length = 20)
    private String documento;

    @NotBlank(message = "El nombre es obligatorio")
    @Size(min = 2, max = 100, message = "El nombre debe tener entre 2 y 100 caracteres")
    @Column(nullable = false, length = 100)
    private String nombre;

    @NotBlank(message = "El email es obligatorio")
    @Email(message = "El formato del email no es válido")
    @Size(max = 150)
    @Column(nullable = false, unique = true, length = 150)
    private String email;

    @Size(max = 20, message = "El teléfono no puede superar 20 caracteres")
    @Pattern(regexp = "^[0-9\\-\\+\\s()]*$",
             message = "El teléfono solo puede contener números y guiones")
    @Column(length = 20)
    private String telefono;    // opcional, pero si se ingresa debe tener formato
}Cliente.java
📄 Turno.java — Validaciones de fecha y relaciones
@Entity @Table(name = "turno")
public class Turno implements Serializable {

    // @NotNull para tipos no-String (LocalDateTime no acepta @NotBlank)
    @NotNull(message = "La fecha y hora del turno son obligatorias")
    // @Future valida automáticamente que la fecha sea futura
    // Complemento: el Servicio también valida esto en código
    @Column(name = "fecha_hora", nullable = false)
    private LocalDateTime fechaHora;

    @Size(max = 200, message = "El motivo no puede superar 200 caracteres")
    @Column(length = 200)
    private String motivo;    // opcional

    // Para relaciones: optional=false actúa como @NotNull en JPA
    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "mascota_id", nullable = false)
    private Mascota mascota;    // siempre debe tener mascota
}Turno.java
💡
Diferencia clave: Usar @NotBlank para campos String (nombre, email, documento) y @NotNull para objetos y tipos primitivos (LocalDateTime, Integer, relaciones JPA). @NotBlank es más estricto: también rechaza cadenas con solo espacios en blanco.

Nivel 3 — Servicio: reglas de negocio
DIAPOSITIVA 9 — ¿Qué valida el Servicio que no puede hacer Bean Validation?
📌
Bean Validation no puede consultar la base de datos. Las reglas que requieren comparar con datos existentes (unicidad, disponibilidad, consistencia entre tablas) siempre deben implementarse en la capa de Servicio.
Regla de negocio¿Puede Bean Validation?¿Dónde implementar?
El email no puede estar duplicadoNo — requiere consulta a BDServicio: buscar por email antes de guardar
El documento debe ser únicoNo — requiere consulta a BDServicio: buscar por documento antes de guardar
La fecha del turno debe ser futura@Future parcialmenteServicio: comparar con LocalDateTime.now()
La mascota debe estar activaNo — requiere consulta a BDServicio: verificar mascota.isActivo()
Un turno REALIZADO no puede cancelarseNo — lógica de estadosServicio: validar el estado actual antes de la transición
El precio no puede ser negativoSí — @DecimalMin("0.00")Bean Validation en la entidad
El nombre no puede estar vacíoSí — @NotBlankBean Validation en la entidad
DIAPOSITIVA 10 — Patrón completo de validación en el Servicio
📄 ClienteServicio.java — Validaciones de negocio
@ApplicationScoped
public class ClienteServicio {

    @Inject private ClienteDAO clienteDAO;

    @Transactional
    public void guardar(Cliente cliente) {

        // ── VALIDACIÓN 1: Documento único ───────────────────────
        clienteDAO.buscarPorDocumento(cliente.getDocumento())
            .ifPresent(existente -> {
                // Es el mismo registro en edición → está OK
                boolean mismoRegistro = existente.getId() != null
                    && existente.getId().equals(cliente.getId());
                if (!mismoRegistro) {
                    // Documento duplicado → lanzar excepción
                    // @Transactional hace ROLLBACK automáticamente
                    throw new IllegalArgumentException(
                        "Ya existe un cliente con el documento: "
                        + cliente.getDocumento()
                    );
                }
            });

        // ── VALIDACIÓN 2: Email único ───────────────────────────
        clienteDAO.buscarPorEmail(cliente.getEmail())
            .ifPresent(existente -> {
                boolean mismoRegistro = existente.getId() != null
                    && existente.getId().equals(cliente.getId());
                if (!mismoRegistro) {
                    throw new IllegalArgumentException(
                        "Ya existe un cliente con el email: "
                        + cliente.getEmail()
                    );
                }
            });

        // ── Si pasan todas las validaciones → persistir ─────────
        if (cliente.getId() == null)
            clienteDAO.guardar(cliente);    // INSERT
        else
            clienteDAO.actualizar(cliente); // UPDATE
    }
}ClienteServicio.java
📄 TurnoServicio.java — Validaciones de estados y fechas
@Transactional
public void reservar(Turno turno, Long mascotaId) {

    // ── VALIDACIÓN 1: Fecha debe ser futura ─────────────────
    if (turno.getFechaHora() == null)
        throw new IllegalArgumentException(
            "La fecha y hora son obligatorias.");

    if (turno.getFechaHora().isBefore(LocalDateTime.now()))
        throw new IllegalArgumentException(
            "No se puede reservar un turno en el pasado.");

    // ── VALIDACIÓN 2: Mascota debe existir y estar activa ────
    Mascota mascota = mascotaDAO.buscarPorId(mascotaId);

    if (mascota == null)
        throw new IllegalArgumentException(
            "La mascota seleccionada no existe.");

    if (!mascota.isActivo())
        throw new IllegalArgumentException(
            "La mascota seleccionada está inactiva. No puede reservar un turno.");

    // ── Si todo está OK → guardar ────────────────────────────
    turno.setMascota(mascota);
    turno.setEstado(Turno.PENDIENTE);
    turnoDAO.guardar(turno);
}

@Transactional
public void confirmar(Long id) {
    Turno turno = turnoDAO.buscarPorId(id);

    // ── VALIDACIÓN 3: Transición de estado válida ────────────
    if (!Turno.PENDIENTE.equals(turno.getEstado())) {
        throw new IllegalStateException(
            "Solo se puede confirmar un turno PENDIENTE. "
            + "Estado actual: " + turno.getEstado()
        );
    }
    turnoDAO.cambiarEstado(id, Turno.CONFIRMADO);
}TurnoServicio.java

Nivel 4 — JPA: restricciones de columna
DIAPOSITIVA 11 — @Column como validación
📄 Restricciones en @Column y @JoinColumn
// nullable=false → genera NOT NULL en la BD
@Column(nullable = false, length = 100)
private String nombre;

// unique=true → genera UNIQUE en la BD
@Column(unique = true, nullable = false, length = 150)
private String email;

// length → VARCHAR(20) en la BD
// Si se supera, MySQL trunca o lanza error
@Column(length = 20)
private String telefono;

// precision y scale para decimales: DECIMAL(10,2)
@Column(precision = 10, scale = 2, nullable = false)
private BigDecimal precio;

// Restricción única por múltiples columnas
@Table(name = "turno",
  uniqueConstraints = @UniqueConstraint(
    columnNames = {"mascota_id", "fecha_hora"}
  )
)
📋 Relación entre @Column y Bean Validation
@ColumnBean Validation equivalente
nullable=false@NotNull / @NotBlank
length=100@Size(max=100)
unique=true⚠️ No tiene equivalente directo
precision=10,scale=2@Digits(integer=8, fraction=2)

Usar ambas en paralelo: Bean Validation actúa antes de ir a la BD (más rápido), y @Column define el esquema real de la base de datos (última barrera).

Nivel 5 — MySQL: restricciones en la BD
DIAPOSITIVA 12 — El script SQL como validación definitiva
📄 veterinariadb.sql — Restricciones MySQL
CREATE TABLE cliente (
    id         BIGINT        NOT NULL AUTO_INCREMENT,

    -- NOT NULL: protege contra valores nulos directos
    documento  VARCHAR(20)  NOT NULL,
    nombre     VARCHAR(100) NOT NULL,
    email      VARCHAR(150) NOT NULL,

    activo     TINYINT(1)   NOT NULL DEFAULT 1,

    -- PRIMARY KEY: unicidad e índice del ID
    CONSTRAINT pk_cliente      PRIMARY KEY (id),

    -- UNIQUE: protege la unicidad de documento y email
    CONSTRAINT uq_cli_doc      UNIQUE (documento),
    CONSTRAINT uq_cli_email    UNIQUE (email)
);

CREATE TABLE mascota (
    id         BIGINT       NOT NULL AUTO_INCREMENT,
    nombre     VARCHAR(80)  NOT NULL,
    especie    VARCHAR(50)  NOT NULL,
    cliente_id BIGINT       NOT NULL,

    CONSTRAINT pk_mascota      PRIMARY KEY (id),

    -- FOREIGN KEY: integridad referencial
    -- ON DELETE RESTRICT: impide eliminar un cliente que tiene mascotas
    CONSTRAINT fk_masc_cliente FOREIGN KEY (cliente_id)
        REFERENCES cliente(id)
        ON DELETE RESTRICT
        ON UPDATE CASCADE
);
✅ ON DELETE RESTRICT (recomendado)

MySQL impide eliminar un cliente que tiene mascotas. Protege la integridad.

ON DELETE RESTRICT
-- Intentar borrar cliente con mascotas
-- → ConstraintViolationException
-- → La app muestra: "Use Desactivar"
⚠️ ON DELETE CASCADE (peligroso)

MySQL elimina automáticamente todas las mascotas al borrar el cliente.

ON DELETE CASCADE
-- Borrar cliente → borra mascotas
-- Borrar mascota → borra turnos
-- Sin confirmación, sin recuperación
-- ❌ No usar en sistemas reales sin respaldo

Mensajes de error al usuario con FacesMessage
DIAPOSITIVA 13 — Los 3 niveles de severidad
ℹ️ SEVERITY_INFO

Cuándo usar: Operación exitosa. Alta, modificación, baja realizadas correctamente.

addMsg(SEVERITY_INFO,
  "Éxito",
  "Cliente guardado.");

Texto en azul o verde en la interfaz.

⚠️ SEVERITY_WARN

Cuándo usar: El usuario ingresó datos que violan una regla de negocio. Ej: email duplicado, fecha en pasado.

addMsg(SEVERITY_WARN,
  "Atención",
  e.getMessage());

Texto en amarillo/naranja.

❌ SEVERITY_ERROR

Cuándo usar: Error técnico inesperado. Fallo de BD, error de red, NPE.

addMsg(SEVERITY_ERROR,
  "Error",
  "No se pudo guardar.");

Texto en rojo. Nunca mostrar el stack trace.

DIAPOSITIVA 14 — Patrón try-catch en el Bean
❌ Sin manejo de errores
public void guardar() {
  // Sin try-catch:
  // Si ocurre error → pantalla de error 500
  // El usuario no sabe qué pasó
  // La aplicación queda en estado inconsistente
  clienteServicio.guardar(clienteSeleccionado);
}
✅ Con manejo de errores correcto
public void guardar() {
  try {
    clienteServicio.guardar(clienteSeleccionado);
    cargarClientes();
    clienteSeleccionado = new Cliente();
    addMsg(SEVERITY_INFO, "Éxito",
           "Cliente guardado.");

  } catch (IllegalArgumentException e) {
    // Error de negocio: mensaje del Servicio
    addMsg(SEVERITY_WARN, "Atención",
           e.getMessage());

  } catch (Exception e) {
    // Error técnico: mensaje genérico
    addMsg(SEVERITY_ERROR, "Error",
           "No se pudo guardar. Intente nuevamente.");
  }
}

DIAPOSITIVA 17 — Validador personalizado en JSF
💡
Cuando los validadores estándar no alcanzan, se puede crear un validador personalizado implementando la interfaz jakarta.faces.validator.Validator. Se registra con @FacesValidator y se usa como cualquier otro validador.
📄 ValidadorDocumento.java — Validador personalizado
import jakarta.faces.validator.FacesValidator;
import jakarta.faces.validator.Validator;
import jakarta.faces.validator.ValidatorException;

// Registrar con un nombre que se usa en la vista
@FacesValidator("validadorDocumento")
public class ValidadorDocumento implements Validator<String> {

    @Override
    public void validate(FacesContext ctx,
                         UIComponent comp,
                         String valor)
                         throws ValidatorException {

        if (valor == null || valor.isBlank()) return;

        // Solo dígitos, entre 6 y 9 caracteres
        if (!valor.matches("^[0-9]{6,9}$")) {
            FacesMessage msg = new FacesMessage(
                FacesMessage.SEVERITY_ERROR,
                "Documento inválido",
                "Solo se permiten entre 6 y 9 dígitos numéricos"
            );
            throw new ValidatorException(msg);
        }
    }
}

Resumen de la clase y actividad práctica
DIAPOSITIVA 19 — Cuadro resumen: los 5 niveles
🌐 Nivel 1 — Vista JSF
Herramientas: required, f:validateRegex, f:validateLength, f:convertDateTime, f:converter

Cuándo actúa: Fase 3 del ciclo JSF (antes de llamar al Bean)

Ventaja: Respuesta inmediata, no va al servidor
Limitación: Se puede saltar con Postman o curl
✅ Nivel 2 — Bean Validation
Herramientas: @NotBlank, @Email, @Size, @Min, @Max, @Pattern, @NotNull

Cuándo actúa: Al persistir con JPA y en Fase 3 JSF

Ventaja: Anotaciones simples en la entidad, reutilizables
Limitación: No puede consultar la BD
⚙️ Nivel 3 — Servicio
Herramientas: Código Java puro + consultas DAO

Cuándo actúa: Al llamar el método de negocio

Ventaja: Puede validar contra la BD (unicidad, existencia, estado)
Limitación: Más código a escribir
🔌 Nivel 4 — JPA
Herramientas: nullable=false, unique=true, length, precision, optional=false

Cuándo actúa: Al hacer persist/merge

Ventaja: Define el esquema de BD y actúa antes del SQL
Limitación: Menos descriptivo que Bean Validation
🗄️ Nivel 5 — MySQL (última barrera)
Herramientas: NOT NULL, UNIQUE, FOREIGN KEY, CHECK, tipos de dato, restricciones de FK

Cuándo actúa: Al ejecutar INSERT/UPDATE en la base de datos

Ventaja: Protege la BD incluso si alguien inserta datos directamente por SQL. No se puede evadir.
Limitación: El mensaje de error no es amigable para el usuario
