5.1 — Patrón DAO (Data Access Object)
🏗️ ¿Qué es el patrón DAO?

El Data Access Object (DAO) es un patrón de diseño que separa la lógica de acceso a datos del resto de la aplicación. Actúa como una capa intermedia entre la lógica de negocio y la base de datos.


En lugar de escribir código SQL o JPA dentro del Bean o el Servicio, todo el acceso a la base de datos queda encapsulado en clases DAO especializadas.

┌─────────────────────────────────────────────────────────────────────┐ │ ARQUITECTURA EN CAPAS │ │ │ │ Vista (.xhtml) │ │ ↕ EL (Expression Language) │ │ ┌─────────────────┐ │ │ │ Managed Bean │ @Named @ViewScoped ← CAPA PRESENTACIÓN │ │ │ (Controlador) │ │ │ └────────┬────────┘ │ │ │ @Inject │ │ ┌────────▼────────┐ │ │ │ Servicio │ @ApplicationScoped ← CAPA DE NEGOCIO │ │ │ (Reglas de │ │ │ │ negocio) │ │ │ └────────┬────────┘ │ │ │ @Inject │ │ ┌────────▼────────┐ │ │ │ DAO │ @ApplicationScoped ← CAPA DE DATOS │ │ │ (Acceso a BD) │ @PersistenceContext │ │ └────────┬────────┘ │ │ │ │ │ ┌────────▼────────┐ │ │ │ EntityManager │ JPA / EclipseLink ← ORM │ │ └────────┬────────┘ │ │ │ JDBC │ │ ┌────────▼────────┐ │ │ │ MySQL 8 │ ← BASE DE DATOS │ │ └─────────────────┘ │ └─────────────────────────────────────────────────────────────────────┘
✅ Responsabilidad del DAO
  • Operaciones CRUD sobre una entidad
  • Consultas JPQL específicas
  • Traducir objetos Java ↔ registros BD
  • Gestión del EntityManager
❌ Lo que el DAO NO debe hacer
  • Contener reglas de negocio
  • Validar datos del formulario
  • Interactuar con la vista
  • Conocer el Bean o la sesión JSF
📦 GenericDAO — DAO Genérico Abstracto
// GenericDAO.java — Clase base abstracta con CRUD genérico
package com.veterinaria.dao;

import jakarta.persistence.EntityManager;
import jakarta.persistence.PersistenceContext;
import jakarta.transaction.Transactional;
import java.util.List;

public abstract class GenericDAO<T, ID> {

    // El contenedor (GlassFish) inyecta el EntityManager JTA
    @PersistenceContext(unitName = "VeterinariaaPU")
    protected EntityManager em;

    // Cada subclase indica su tipo de entidad
    protected abstract Class<T> getEntityClass();

    @Transactional
    public void guardar(T entidad)     { em.persist(entidad); }

    @Transactional
    public T actualizar(T entidad)     { return em.merge(entidad); }

    @Transactional
    public void eliminar(ID id) {
        T entidad = em.find(getEntityClass(), id);
        if (entidad != null) em.remove(entidad);
    }

    public T buscarPorId(ID id) {
        return em.find(getEntityClass(), id);
    }

    public List<T> listarTodos() {
        String jpql = "SELECT e FROM "
                    + getEntityClass().getSimpleName() + " e";
        return em.createQuery(jpql, getEntityClass()).getResultList();
    }
}GenericDAO.java
💡
Patrón Genérico: Usando Generics (<T, ID>), el GenericDAO sirve para cualquier entidad. ClienteDAO extiende GenericDAO<Cliente, Long> y hereda automáticamente los 5 métodos base.

5.2 — EntityManager: Métodos CRUD
⚙️ ¿Qué es el EntityManager?

El EntityManager es la interfaz central de JPA. Es el objeto que conecta el mundo Java con la base de datos. Permite persistir, actualizar, eliminar y buscar entidades sin escribir SQL directamente.


En Jakarta EE con GlassFish, el contenedor inyecta el EntityManager automáticamente usando @PersistenceContext. No necesitamos crearlo ni cerrarlo manualmente.

Método SQL equivalente Descripción Necesita @Transactional
em.persist(entidad) INSERT Inserta una entidad nueva en la BD Sí
em.merge(entidad) UPDATE Actualiza una entidad existente Sí
em.remove(entidad) DELETE Elimina una entidad de la BD Sí
em.find(Clase.class, id) SELECT por PK Busca una entidad por clave primaria No
em.createQuery(jpql) SELECT personalizado Ejecuta consulta JPQL No
em.createNamedQuery(nombre) SELECT predefinido Ejecuta una NamedQuery definida en la entidad No
em.flush() — Sincroniza el contexto de persistencia con la BD Sí
em.refresh(entidad) SELECT recarga Recarga el estado desde la BD, descarta cambios No
em.detach(entidad) — Desasocia la entidad del contexto de persistencia No
🔵 persist — INSERT
// Solo para entidades NUEVAS (id == null)
Cliente c = new Cliente();
c.setNombre("María");
c.setEmail("maria@mail.com");

// genera: INSERT INTO cliente (...) VALUES (...)
em.persist(c);

// Después del persist, c.getId() ya tiene valor
System.out.println(c.getId()); // ej: 1
🟡 merge — UPDATE
// Para entidades EXISTENTES (id != null)
// El objeto puede estar "detached" (fuera del contexto)
Cliente c = em.find(Cliente.class, 1L);
c.setTelefono("11-9999-8888");

// genera: UPDATE cliente SET telefono=? WHERE id=?
Cliente managed = em.merge(c);

// merge devuelve la instancia gestionada actualizada
🔴 remove — DELETE
// remove solo acepta entidades MANAGED
// Primero hay que hacer find, luego remove
Cliente c = em.find(Cliente.class, 1L);

if (c != null) {
    // genera: DELETE FROM cliente WHERE id=1
    em.remove(c);
}
// Si id no existe, find devuelve null (no da error)
🟢 find — SELECT por PK
// Busca por clave primaria — el más eficiente
// Usa el caché de primer nivel si ya fue cargado
Cliente c = em.find(Cliente.class, 5L);

// genera: SELECT * FROM cliente WHERE id=5

if (c == null) {
    // El cliente con id=5 no existe en la BD
    System.out.println("No encontrado");
}
⚠️
Error común: Llamar a em.remove(entidad) con una entidad detached lanza una excepción. Siempre hay que hacer em.find() primero para obtener la entidad managed, y luego pasársela a em.remove().
🔄 Estados de una Entidad JPA
NEW (nueva) → persist() → MANAGED (gestionada) new Cliente() em tiene referencia activa │ │ │ │ detach() remove() │ │ │ └──────────────── merge() ──── DETACHED REMOVED (fuera del (marcada contexto) para DELETE)
  • NEW: objeto Java recién creado, no conocido por JPA
  • MANAGED: JPA rastrea sus cambios — cualquier setter genera un UPDATE al hacer flush/commit
  • DETACHED: fue managed pero se desconectó (fin de transacción, detach())
  • REMOVED: marcada para eliminar — el DELETE ocurre al hacer flush/commit

5.3 — JPQL: Jakarta Persistence Query Language
🔍 ¿Qué es JPQL?

JPQL es el lenguaje de consultas de JPA. Es similar a SQL pero trabaja sobre entidades Java, no sobre tablas. EclipseLink traduce el JPQL al SQL específico del motor de BD (MySQL en nuestro caso).


Diferencia fundamental: En SQL se usan nombres de tablas y columnas. En JPQL se usan nombres de clases y atributos Java.

SQLJPQL equivalente
SELECT * FROM cliente SELECT c FROM Cliente c
SELECT * FROM cliente WHERE id = 5 SELECT c FROM Cliente c WHERE c.id = 5
SELECT * FROM cliente WHERE activo = 1 ORDER BY apellido SELECT c FROM Cliente c WHERE c.activo = true ORDER BY c.apellido
SELECT * FROM cliente WHERE email = ? SELECT c FROM Cliente c WHERE c.email = :email
SELECT * FROM mascota WHERE cliente_id = ? SELECT m FROM Mascota m WHERE m.cliente.id = :id
SELECT COUNT(*) FROM cliente SELECT COUNT(c) FROM Cliente c

▶ Tipos de consultas JPQL

1️⃣ createQuery — consulta dinámica
// TypedQuery garantiza el tipo de retorno en tiempo de compilación
String jpql = "SELECT c FROM Cliente c WHERE c.activo = true ORDER BY c.apellido";

TypedQuery<Cliente> query = em.createQuery(jpql, Cliente.class);

List<Cliente> lista = query.getResultList();   // devuelve lista
Cliente uno     = query.getSingleResult();     // devuelve uno solo
2️⃣ Parámetros nombrados con :nombre
// Los parámetros nombrados evitan SQL injection
String jpql = "SELECT c FROM Cliente c WHERE c.email = :email";

List<Cliente> resultado = em
    .createQuery(jpql, Cliente.class)
    .setParameter("email", "maria@mail.com") // :email ← valor
    .getResultList();

// Búsqueda parcial con LIKE (case insensitive)
String busqueda = "SELECT c FROM Cliente c WHERE LOWER(c.nombre) LIKE :patron";
em.createQuery(busqueda, Cliente.class)
  .setParameter("patron", "%" + texto.toLowerCase() + "%")
  .getResultList();
3️⃣ @NamedQuery — consulta predefinida en la entidad
// Se define UNA VEZ en la clase de entidad
@Entity
@NamedQueries({
    @NamedQuery(
        name  = "Cliente.findAll",
        query = "SELECT c FROM Cliente c ORDER BY c.apellido, c.nombre"
    ),
    @NamedQuery(
        name  = "Cliente.findActivos",
        query = "SELECT c FROM Cliente c WHERE c.activo = true ORDER BY c.apellido"
    ),
    @NamedQuery(
        name  = "Cliente.findByEmail",
        query = "SELECT c FROM Cliente c WHERE c.email = :email"
    )
})
public class Cliente { ... }

// Se usa en el DAO así:
em.createNamedQuery("Cliente.findAll", Cliente.class)
  .getResultList();

em.createNamedQuery("Cliente.findByEmail", Cliente.class)
  .setParameter("email", emailBuscado)
  .getResultList();
4️⃣ JOIN FETCH — evitar el problema N+1
// ❌ SIN JOIN FETCH: genera 1 + N consultas
// 1 SELECT para mascotas + 1 SELECT por cada mascota para cargar cliente
"SELECT m FROM Mascota m"  // si hay 100 mascotas → 101 queries!

// ✅ CON JOIN FETCH: genera UNA SOLA consulta con JOIN
"SELECT m FROM Mascota m JOIN FETCH m.cliente ORDER BY m.nombre"
// → SELECT m.*, c.* FROM mascota m INNER JOIN cliente c ON m.cliente_id = c.id
✅
Buena práctica: Usar siempre @NamedQuery para las consultas frecuentes. Se validan al arrancar el servidor y se compilan una sola vez, siendo más eficientes que las consultas dinámicas.

5.4 — Transacciones con @Transactional
🔒 ¿Qué es una transacción?

Una transacción es un conjunto de operaciones sobre la base de datos que se ejecutan como una unidad: todas tienen éxito o ninguna se aplica. Garantiza la integridad de los datos.


En Jakarta EE con JTA (Jakarta Transactions API), el servidor GlassFish gestiona las transacciones automáticamente. Solo necesitamos indicar con @Transactional qué métodos deben ejecutarse dentro de una transacción.

✅ Las 4 propiedades ACID
PropiedadSignificadoEjemplo
AtomicidadTodo o nadaSi falla el INSERT, no queda nada a medias
ConsistenciaLa BD pasa de un estado válido a otroNo puede quedar un turno sin mascota
AislamientoTransacciones no se interfierenDos usuarios no editan el mismo registro simultáneamente
DurabilidadLos cambios son permanentesUn COMMIT sobrevive a un corte de luz

▶ @Transactional en la práctica

Comportamiento de @Transactional
@ApplicationScoped
public class ClienteServicio {

    @Inject
    private ClienteDAO clienteDAO;

    // ✅ @Transactional: el contenedor abre TX antes y hace COMMIT al final
    // Si se lanza una RuntimeException → ROLLBACK automático
    @Transactional
    public void guardar(Cliente cliente) {

        // 1. Validación de negocio (antes de tocar la BD)
        clienteDAO.buscarPorEmail(cliente.getEmail())
            .ifPresent(existente -> {
                boolean mismo = existente.getId().equals(cliente.getId());
                if (!mismo)
                    // RuntimeException → ROLLBACK automático
                    throw new IllegalArgumentException("Email duplicado");
            });

        // 2. Alta o modificación
        if (cliente.getId() == null)
            clienteDAO.guardar(cliente);   // INSERT
        else
            clienteDAO.actualizar(cliente); // UPDATE

    } // ← COMMIT aquí (si no hubo excepción)

    // Sin @Transactional → solo lectura, más eficiente
    public List<Cliente> listarTodos() {
        return clienteDAO.listarTodos();
    }
}
✅ Con @Transactional (escritura)
Llamada al método ↓ GlassFish abre TX ↓ persist / merge / remove ↓ ¿Excepción? ├── NO → COMMIT ✅ └── SÍ → ROLLBACK ❌
❌ Sin @Transactional (en escritura)
Llamada al método ↓ persist / merge / remove ↓ TransactionRequiredException "No active transaction" ↓ 💥 ERROR en runtime
⚠️
Regla importante: persist(), merge() y remove() siempre necesitan @Transactional. Los métodos de solo lectura (find(), getResultList()) no lo necesitan, aunque no rompen si lo tienen.
📋 Tipos de propagación @Transactional
import jakarta.transaction.Transactional;
import jakarta.transaction.Transactional.TxType;

// REQUIRED (defecto): usa TX existente o crea una nueva
@Transactional(TxType.REQUIRED)
public void guardar(Cliente c) { ... }

// REQUIRES_NEW: siempre crea una TX nueva, suspende la existente
@Transactional(TxType.REQUIRES_NEW)
public void auditoria(String accion) { ... }

// NOT_SUPPORTED: se ejecuta sin TX (lectura pura, más eficiente)
@Transactional(TxType.NOT_SUPPORTED)
public List<Cliente> listarTodos() { ... }

// NEVER: lanza excepción si existe TX activa
@Transactional(TxType.NEVER)
public void soloLectura() { ... }

📄 ClienteDAO.java — Completo
package com.veterinaria.dao;

import com.veterinaria.modelo.Cliente;
import jakarta.enterprise.context.ApplicationScoped;
import jakarta.transaction.Transactional;
import java.util.List;
import java.util.Optional;

/**
 * DAO para la entidad Cliente.
 * Extiende GenericDAO y agrega consultas específicas del dominio.
 *
 * @ApplicationScoped → CDI crea UNA sola instancia para toda la app
 */
@ApplicationScoped
public class ClienteDAO extends GenericDAO<Cliente, Long> {

    @Override
    protected Class<Cliente> getEntityClass() { return Cliente.class; }

    // ─── Consultas específicas ──────────────────────────────

    /** Todos los clientes ordenados por apellido y nombre */
    public List<Cliente> listarTodos() {
        return em.createNamedQuery("Cliente.findAll", Cliente.class)
                 .getResultList();
    }

    /** Solo clientes con activo = true */
    public List<Cliente> listarActivos() {
        return em.createNamedQuery("Cliente.findActivos", Cliente.class)
                 .getResultList();
    }

    /** Busca por número de documento exacto — retorna Optional */
    public Optional<Cliente> buscarPorDocumento(String documento) {
        List<Cliente> r = em
            .createNamedQuery("Cliente.findByDocumento", Cliente.class)
            .setParameter("documento", documento)
            .getResultList();
        return r.isEmpty() ? Optional.empty() : Optional.of(r.get(0));
    }

    /** Busca por email exacto — retorna Optional */
    public Optional<Cliente> buscarPorEmail(String email) {
        List<Cliente> r = em
            .createNamedQuery("Cliente.findByEmail", Cliente.class)
            .setParameter("email", email)
            .getResultList();
        return r.isEmpty() ? Optional.empty() : Optional.of(r.get(0));
    }

    /** Búsqueda parcial por nombre, apellido o documento */
    public List<Cliente> buscarPorTexto(String texto) {
        String patron = "%" + texto.toLowerCase().trim() + "%";
        return em
            .createNamedQuery("Cliente.buscarPorTexto", Cliente.class)
            .setParameter("patron", patron)
            .getResultList();
    }

    /** Baja lógica: activo = false, sin eliminar el registro */
    @Transactional
    public void desactivar(Long id) {
        Cliente c = em.find(Cliente.class, id);
        if (c != null) {
            c.setActivo(false);
            em.merge(c);
        }
    }

    /** Reactiva un cliente previamente desactivado */
    @Transactional
    public void activar(Long id) {
        Cliente c = em.find(Cliente.class, id);
        if (c != null) {
            c.setActivo(true);
            em.merge(c);
        }
    }
}ClienteDAO.java
🔎
¿Por qué Optional? Cuando buscamos por email o documento esperamos 0 o 1 resultado. Retornar Optional<Cliente> en lugar de Cliente o null obliga al llamador a manejar explícitamente el caso "no encontrado", evitando NullPointerException.

📄 ClienteBean.java — Versión completa con DAO
package com.veterinaria.controlador;

import com.veterinaria.modelo.Cliente;
import com.veterinaria.servicio.ClienteServicio;
import jakarta.annotation.PostConstruct;
import jakarta.faces.application.FacesMessage;
import jakarta.faces.context.FacesContext;
import jakarta.faces.view.ViewScoped;
import jakarta.inject.Inject;
import jakarta.inject.Named;
import java.io.Serializable;
import java.util.List;

@Named("clienteBean")
@ViewScoped  // vive mientras el usuario está en la misma vista
public class ClienteBean implements Serializable {

    private static final long serialVersionUID = 1L;

    // ── CDI inyecta el servicio ──────────────────────────────
    @Inject
    private ClienteServicio clienteServicio;

    // ── Estado de la vista ───────────────────────────────────
    private List<Cliente> clientes;
    private Cliente        clienteSeleccionado;
    private String         textoBusqueda;
    private boolean        modoEdicion = false;

    // ── @PostConstruct: se ejecuta al crear el bean ──────────
    @PostConstruct
    public void init() {
        clienteSeleccionado = new Cliente();
        cargarClientes();  // carga desde BD al iniciar
    }

    // ── Carga de datos desde BD ──────────────────────────────
    public void cargarClientes() {
        clientes = clienteServicio.listarTodos();
    }

    // ── Preparar formulario para nueva alta ──────────────────
    public void prepararNuevo() {
        clienteSeleccionado = new Cliente();
        modoEdicion = false;
    }

    // ── Cargar cliente existente en el formulario ────────────
    public void prepararEdicion(Cliente c) {
        // Recargamos desde BD para tener datos frescos
        clienteSeleccionado = clienteServicio.buscarPorId(c.getId());
        modoEdicion = true;
    }

    // ── Guardar (INSERT o UPDATE) ────────────────────────────
    public void guardar() {
        try {
            clienteServicio.guardar(clienteSeleccionado); // llama al DAO
            cargarClientes();                           // refresca la tabla
            clienteSeleccionado = new Cliente();
            modoEdicion = false;
            addMsg(FacesMessage.SEVERITY_INFO,
                "Éxito", "Cliente guardado correctamente.");

        } catch (IllegalArgumentException e) {
            addMsg(FacesMessage.SEVERITY_WARN, "Atención", e.getMessage());

        } catch (Exception e) {
            addMsg(FacesMessage.SEVERITY_ERROR,
                "Error", "No se pudo guardar el cliente.");
        }
    }

    // ── Eliminar (DELETE físico) ─────────────────────────────
    public void eliminar(Cliente c) {
        try {
            clienteServicio.eliminar(c.getId());
            cargarClientes();
            addMsg(FacesMessage.SEVERITY_INFO,
                "Eliminado", "El cliente fue eliminado.");
        } catch (Exception e) {
            addMsg(FacesMessage.SEVERITY_ERROR, "Error",
                "El cliente tiene mascotas. Use Desactivar.");
        }
    }

    // ── Desactivar (baja lógica) ─────────────────────────────
    public void desactivar(Cliente c) {
        clienteServicio.desactivar(c.getId());
        cargarClientes();
        addMsg(FacesMessage.SEVERITY_INFO,
            "Desactivado", c.getNombreCompleto() + " fue desactivado.");
    }

    // ── Búsqueda ─────────────────────────────────────────────
    public void buscar() {
        clientes = clienteServicio.buscar(textoBusqueda);
    }

    // ── Cancelar edición ─────────────────────────────────────
    public void cancelar() {
        clienteSeleccionado = new Cliente();
        textoBusqueda = null;
        modoEdicion = false;
        cargarClientes();
    }

    // ── Helper para mensajes JSF ─────────────────────────────
    private void addMsg(FacesMessage.Severity s, String r, String d) {
        FacesContext.getCurrentInstance()
            .addMessage(null, new FacesMessage(s, r, d));
    }

    // ── Getters y Setters ────────────────────────────────────
    public List<Cliente> getClientes()            { return clientes; }
    public Cliente getClienteSeleccionado()        { return clienteSeleccionado; }
    public void setClienteSeleccionado(Cliente c) { clienteSeleccionado = c; }
    public String getTextoBusqueda()              { return textoBusqueda; }
    public void setTextoBusqueda(String t)        { textoBusqueda = t; }
    public boolean isModoEdicion()                { return modoEdicion; }
}ClienteBean.java

Resumen y Puntos Clave
🏗️ Patrón DAO
  • Separa el acceso a datos del negocio
  • GenericDAO<T,ID> provee CRUD base
  • Cada entidad tiene su DAO específico
  • @ApplicationScoped = una instancia
⚙️ EntityManager
  • @PersistenceContext para inyectarlo
  • persist → INSERT (entidad nueva)
  • merge → UPDATE (entidad existente)
  • remove → DELETE (después de find)
  • find → SELECT por PK
🔍 JPQL
  • Trabaja con clases Java, no con tablas SQL
  • :parametro para evitar injection
  • @NamedQuery para consultas frecuentes
  • JOIN FETCH evita el problema N+1
🔒 @Transactional
  • Obligatorio en persist, merge, remove
  • Commit automático al terminar el método
  • Rollback automático ante RuntimeException
  • Lecturas no lo necesitan (pero no rompen)

🎯
Logro de la clase: Reemplazamos el ArrayList temporal por una arquitectura profesional de 4 capas conectada a MySQL. El Bean ya no sabe nada de SQL: delega todo al Servicio que delega al DAO que usa el EntityManager. Cada capa tiene una sola responsabilidad — esto es lo que hace un sistema mantenible y escalable.
💻 Actividad Práctica — Clase 05
1

Crear GenericDAO.java

Crear el paquete com.veterinaria.dao y la clase abstracta genérica con los 5 métodos base.

2

Agregar @NamedQueries a Cliente.java

Definir las 5 NamedQueries en la entidad: findAll, findActivos, findByDocumento, findByEmail y buscarPorTexto.

3

Crear ClienteDAO.java

Extender GenericDAO y agregar los métodos específicos: listarTodos, buscarPorEmail, buscarPorTexto, desactivar, activar.

4

Crear ClienteServicio.java

Inyectar ClienteDAO con @Inject y agregar las reglas de negocio: validar email único, validar documento único.

5

Actualizar ClienteBean.java

Reemplazar el ArrayList por @Inject ClienteServicio. Actualizar todos los métodos para usar el servicio. Cambiar @SessionScoped a @ViewScoped.

6

Probar el CRUD completo

Alta, búsqueda, modificación, desactivar y eliminar. Verificar en MySQL que los cambios persisten al recargar la página.
