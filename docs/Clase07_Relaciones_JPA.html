1
¿Qué es una relación en JPA?
DIAPOSITIVA 1 — Concepto fundamental
🔗 Base de datos vs Mundo Orientado a Objetos

En una base de datos relacional, las tablas se vinculan mediante claves foráneas (FK). En Java, los objetos se vinculan mediante referencias a otros objetos. JPA es el puente que traduce un mundo al otro automáticamente.

🗄️ MySQL — Clave Foránea
CREATE TABLE mascota (
  id         BIGINT PRIMARY KEY,
  nombre     VARCHAR(80),
  cliente_id BIGINT NOT NULL,
  FOREIGN KEY (cliente_id)
    REFERENCES cliente(id)
);

-- Para obtener mascota CON su dueño:
SELECT m.nombre, c.apellido
FROM mascota m
INNER JOIN cliente c
  ON m.cliente_id = c.id;
☕ JPA — Referencia a objeto
@Entity
public class Mascota {
  @Id private Long id;
  private String nombre;

  // No hay "cliente_id" — hay un objeto Cliente
  @ManyToOne
  @JoinColumn(name = "cliente_id")
  private Cliente cliente; // ← objeto Java
}

// Para obtener mascota CON su dueño:
mascota.getCliente().getApellido()

Tipos de relaciones JPA
DIAPOSITIVA 3 — Las 4 anotaciones de relación
AnotaciónCardinalidad¿Dónde va la FK?Ejemplo del mundo real
@ManyToOne Muchos → Uno En la tabla del lado Muchos Muchas Mascotas → un Cliente
@OneToMany Uno → Muchos En la tabla del lado Muchos (igual que @ManyToOne) Un Cliente → muchas Mascotas
@OneToOne Uno → Uno En cualquiera de las dos tablas Un Usuario → un Perfil
@ManyToMany Muchos → Muchos Tabla intermedia (JPA la crea automáticamente) Muchos Médicos ↔ Muchos Pacientes
DIAPOSITIVA 4 — Unidireccional vs Bidireccional
➡️ Unidireccional — solo un lado conoce la relación
// UNIDIRECCIONAL:
// Solo Mascota conoce a Cliente
// Cliente no sabe que tiene mascotas

public class Mascota {
  @ManyToOne
  private Cliente cliente; // ✅ conoce al cliente
}

public class Cliente {
  // ❌ No tiene List<Mascota>
  // No puede navegar cliente.getMascotas()
}

Más simple. Usar cuando no necesitamos navegar desde el lado "uno".

↔️ Bidireccional — ambos lados se conocen
// BIDIRECCIONAL:
// Ambas clases se conocen entre sí

public class Mascota {
  @ManyToOne
  private Cliente cliente; // lado dueño (tiene FK)
}

public class Cliente {
  @OneToMany(mappedBy = "cliente")
  private List<Mascota> mascotas;
  // ✅ puede: cliente.getMascotas()
}

Más útil. Permite navegar en ambas direcciones con getters.

⚠️
Clave del @OneToMany bidireccional: el atributo mappedBy es obligatorio y debe contener el nombre del campo en la clase del lado "Muchos" que tiene la FK. En el ejemplo: mappedBy = "cliente" porque en Mascota el campo se llama cliente.
3
@ManyToOne — el lado de la clave foránea
DIAPOSITIVA 5 — @ManyToOne: la anotación más usada
📌
Regla de oro: @ManyToOne siempre va en la clase que tiene la clave foránea en su tabla. Mascota tiene cliente_id → @ManyToOne va en Mascota.
📄 Mascota.java — @ManyToOne completo con todos los atributos
@Entity
@Table(name = "mascota")
public class Mascota implements Serializable {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String nombre;
    private String especie;

    // ═══ RELACIÓN @ManyToOne ══════════════════════════════════
    @ManyToOne(
        fetch    = FetchType.LAZY,   // no carga cliente hasta que se acceda
        optional = false            // = NOT NULL — mascota DEBE tener cliente
    )
    @JoinColumn(
        name             = "cliente_id", // nombre de la columna FK en tabla mascota
        nullable         = false,       // restricción en la BD
        foreignKey       = @ForeignKey(name = "fk_mascota_cliente")
    )
    private Cliente cliente;
    // ══════════════════════════════════════════════════════════
    // EclipseLink genera en MySQL:
    // FOREIGN KEY (cliente_id) REFERENCES cliente(id)

    // Getters y Setters
    public Cliente getCliente()             { return cliente; }
    public void    setCliente(Cliente c)    { this.cliente = c; }

    // Helper útil para la vista
    public String getNombreConDueno() {
        return nombre + " (" + cliente.getApellido() + ")";
    }
}Mascota.java
DIAPOSITIVA 6 — Atributos de @ManyToOne y @JoinColumn
AtributoAnotaciónValores posiblesDescripción
fetch@ManyToOneLAZY / EAGER¿Cuándo cargar el objeto relacionado?
optional@ManyToOnetrue / false¿Puede ser null? false = NOT NULL
cascade@ManyToOneCascadeType.ALL, PERSIST…¿Propagar operaciones al objeto relacionado?

✅
Por defecto: @ManyToOne sin ningún atributo es equivalente a @ManyToOne(fetch=EAGER, optional=true). En aplicaciones profesionales siempre conviene especificar fetch=LAZY y optional=false cuando el campo no puede ser nulo.
4
@OneToMany — el lado inverso (sin FK)
DIAPOSITIVA 7 — @OneToMany: la colección del lado "uno"
📌
Importante: @OneToMany no crea ninguna columna nueva en la tabla cliente. Solo le dice a JPA que puede navegar la relación en sentido inverso. La FK sigue siendo cliente_id en la tabla mascota.
📄 Cliente.java — @OneToMany con todos los atributos explicados
@Entity
@Table(name = "cliente")
public class Cliente implements Serializable {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String nombre;
    private String email;

    // ═══ RELACIÓN @OneToMany ════════════════════════════════
    @OneToMany(
        mappedBy      = "cliente",   // campo en Mascota que tiene la FK
        cascade       = CascadeType.ALL, // propaga persist/merge/remove
        fetch         = FetchType.LAZY,  // no carga lista hasta que se acceda
        orphanRemoval = true           // elimina mascotas sin cliente
    )
    private List<Mascota> mascotas = new ArrayList<>();
    // ════════════════════════════════════════════════════════
    // NO genera ninguna columna en tabla cliente
    // La FK cliente_id ya existe en tabla mascota
    // Permite: cliente.getMascotas()  ← navegar la relación

    public List<Mascota> getMascotas()          { return mascotas; }
    public void           setMascotas(List<Mascota> m) { this.mascotas = m; }

    // Helper: cantidad de mascotas activas
    public long getCantidadMascotas() {
        return mascotas.stream()
                       .filter(Mascota::isActivo)
                       .count();
    }
}Cliente.java
DIAPOSITIVA 8 — Atributos de @OneToMany
AtributoTipoDescripción¿Cuándo usarlo?
mappedBy String Nombre del campo en la entidad "Muchos" que tiene la FK. Obligatorio en bidireccional. Siempre en relación bidireccional
cascade CascadeType[] Operaciones que se propagan a los hijos (PERSIST, MERGE, REMOVE, ALL) Cuando los hijos son "propiedad" del padre
fetch FetchType LAZY (por defecto en @OneToMany) o EAGER LAZY siempre salvo justificación de performance
orphanRemoval boolean Si un hijo se quita de la lista, JPA lo elimina de la BD Relaciones padre-hijo fuertes (Turno → Tratamiento)

Relación bidireccional completa
DIAPOSITIVA 9 — Las 3 entidades juntas
📄 Turno.java — @ManyToOne con Mascota
@Entity
@Table(name = "turno")
public class Turno implements Serializable {

    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private LocalDateTime fechaHora;
    private String motivo;
    private String estado = "PENDIENTE";

    // FK en tabla turno → mascota_id
    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "mascota_id", nullable = false)
    private Mascota mascota;

    // Getters / Setters
    public Mascota getMascota()           { return mascota; }
    public void    setMascota(Mascota m)   { this.mascota = m; }

    // Navegación de 3 niveles: turno → mascota → cliente
    public String getNombreDueno() {
        return mascota.getCliente().getNombreCompleto();
    }
}Turno.java
📄 Mascota.java — Actualizada con @OneToMany hacia Turno
@Entity
public class Mascota implements Serializable {

    // ... campos id, nombre, especie, raza ...

    // Relación hacia arriba: Mascota → Cliente (tiene FK)
    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "cliente_id", nullable = false)
    private Cliente cliente;

    // Relación hacia abajo: Mascota → Turnos (inversa, sin FK)
    @OneToMany(
        mappedBy      = "mascota",       // campo en Turno
        cascade       = CascadeType.ALL,
        fetch         = FetchType.LAZY,
        orphanRemoval = true
    )
    private List<Turno> turnos = new ArrayList<>();
    // Mascota es "nodo intermedio": tiene FK hacia arriba
    // y lista de hijos hacia abajo
}Mascota.java
✅
Resumen de quién tiene qué:
• Tabla mascota → columna cliente_id (FK) → anotación @ManyToOne en clase Mascota
• Tabla turno → columna mascota_id (FK) → anotación @ManyToOne en clase Turno
• Clase Cliente → @OneToMany(mappedBy="cliente") para navegar la colección
• Clase Mascota → @OneToMany(mappedBy="mascota") para navegar los turnos
6
FetchType: LAZY vs EAGER
DIAPOSITIVA 10 — ¿Cuándo carga JPA los datos relacionados?
FetchType.LAZY — Carga bajo demanda
LAZY
// Paso 1: solo carga la mascota Mascota m = em.find(Mascota.class, 1L); // → SELECT id, nombre, especie FROM mascota WHERE id=1 // (cliente NO se carga todavía) // Paso 2: acceder al cliente DISPARA un SELECT adicional String apellido = m.getCliente().getApellido(); // → SELECT id, nombre, apellido, email FROM cliente WHERE id=? // (se ejecuta en este momento, solo si se necesita)
FetchType.EAGER — Carga inmediata con JOIN
EAGER
// Carga la mascota Y el cliente en una sola query con JOIN: Mascota m = em.find(Mascota.class, 1L); // → SELECT m.id, m.nombre, m.especie, c.id, c.nombre, c.apellido // FROM mascota m LEFT JOIN cliente c ON m.cliente_id = c.id // WHERE m.id = 1 // (siempre hace el JOIN, lo uses o no)
DIAPOSITIVA 11 — FetchType por defecto según anotación
AnotaciónFetchType por defectoRecomendación
@ManyToOne EAGER ⚠️ Cambiar a LAZY para mejorar performance
@OneToOne EAGER ⚠️ Cambiar a LAZY para mejorar performance
@OneToMany LAZY ✅ Correcto por defecto
@ManyToMany LAZY ✅ Correcto por defecto
✅ Cuándo usar LAZY
  • Listados donde no siempre se muestra la relación
  • Entidades con muchas relaciones
  • Aplicaciones con alto volumen de datos
  • Regla general: empezar siempre con LAZY
⚠️ Cuándo puede convenir EAGER
  • Relaciones pequeñas que siempre se usan
  • Entidades que siempre se muestran con su relación
  • Cuando ya se resolvió el N+1 de otra forma
  • Nunca EAGER en colecciones (@OneToMany)

El problema N+1 y la solución JOIN FETCH
DIAPOSITIVA 12 — ¿Qué es el problema N+1?
❌ Problema N+1: 100 mascotas = 101 consultas SQL
// Esta consulta JPQL con LAZY carga las mascotas...
List<Mascota> mascotas =
    em.createQuery("SELECT m FROM Mascota m", Mascota.class)
      .getResultList();

// → SQL #1: SELECT id, nombre, especie, cliente_id FROM mascota
//   Resultado: 100 mascotas en memoria

// Ahora en el XHTML se muestra: #{m.cliente.apellido}
// Por CADA mascota JSF llama m.getCliente() → JPA hace 1 SELECT
// → SQL #2:  SELECT * FROM cliente WHERE id=1
// → SQL #3:  SELECT * FROM cliente WHERE id=3
// → SQL #4:  SELECT * FROM cliente WHERE id=1  (¡mismo id, repetido!)
// → SQL #5:  SELECT * FROM cliente WHERE id=7
// ... 100 SELECT más = 101 consultas en total ❌

// Con 1000 mascotas → 1001 consultas ❌ Performance desastrosa
DIAPOSITIVA 13 — Solución: JOIN FETCH en JPQL
✅ JOIN FETCH: una sola consulta con JOIN
// JOIN FETCH fuerza la carga del cliente en la misma consulta
List<Mascota> mascotas =
    em.createQuery(
        "SELECT m FROM Mascota m JOIN FETCH m.cliente ORDER BY m.nombre",
        Mascota.class
    ).getResultList();

// → SQL generado (1 sola consulta):
// SELECT m.id, m.nombre, m.especie, m.raza,
//        c.id, c.nombre, c.apellido, c.email
// FROM mascota m
// INNER JOIN cliente c ON m.cliente_id = c.id
// ORDER BY m.nombre
//
// 100 mascotas → 1 consulta ✅  1000 mascotas → 1 consulta ✅

// Para 3 niveles (Turno → Mascota → Cliente):
em.createQuery(
    "SELECT t FROM Turno t "
  + "JOIN FETCH t.mascota m "
  + "JOIN FETCH m.cliente "
  + "ORDER BY t.fechaHora DESC",
    Turno.class
).getResultList();
// → 1 query con 2 JOINs: turno → mascota → cliente ✅
DIAPOSITIVA 14 — JOIN FETCH en @NamedQuery
📄 Turno.java — NamedQueries con JOIN FETCH
@Entity
@NamedQueries({

              + "JOIN FETCH m.cliente "

        query = "SELECT t FROM Turno t "

              + "JOIN FETCH m.cliente "

})

Regla práctica: Siempre que una lista se muestre en la vista y acceda a propiedades de relaciones LAZY, usar JOIN FETCH en la consulta del DAO. Esto evita el problema N+1 y hace la aplicación mucho más eficiente.

CascadeType: propagar operaciones
DIAPOSITIVA 15 — Tabla completa de CascadeType
CascadeTypePropagaEjemplo práctico¿Cuándo usar?
PERSISTem.persist()Guardar Cliente guarda sus mascotas nuevasCuando se crean hijos desde el padre
MERGEem.merge()Actualizar Cliente actualiza sus mascotasCuando se editan hijos desde el padre
REMOVEem.remove()Eliminar Cliente elimina sus mascotas⚠️ Solo si los hijos no tienen otras FK
REFRESHem.refresh()Recargar Cliente recarga sus mascotasCuando se sincroniza con la BD
DETACHem.detach()Desconectar Cliente desconecta sus mascotasRaro, uso avanzado
ALLTodasPropaga todas las operacionesRelación fuerte padre-hijo
DIAPOSITIVA 16 — CascadeType en la práctica
✅ CascadeType.ALL: relación fuerte
// Cliente "posee" a sus mascotas
// Si se elimina el cliente → se eliminan sus mascotas
@OneToMany(
    mappedBy      = "cliente",
    cascade       = CascadeType.ALL, // todo en cascada
    orphanRemoval = true            // quitar de lista = DELETE
)
private List<Mascota> mascotas;

// Al agregar mascota a la lista y guardar el cliente:
Cliente c = em.find(Cliente.class, 1L);
Mascota nueva = new Mascota("Rex", "Perro");
nueva.setCliente(c);                  // ← siempre setear ambos lados
c.getMascotas().add(nueva);           // ← agregar a la colección
// em.persist(nueva) ← NO necesario si hay CascadeType.PERSIST
⚠️ Sin cascade: relación débil
// Turno referencia a Mascota,
// pero NO "posee" a Mascota
@ManyToOne(
    fetch    = FetchType.LAZY,
    optional = false
    // sin cascade: eliminar Turno NO afecta a Mascota
)
@JoinColumn(name = "mascota_id")
private Mascota mascota;

// ⚠️ Si se intenta eliminar Mascota que tiene Turnos:
// → ConstraintViolationException
// Solución: baja lógica (activo=false) en lugar de DELETE
⚠️
Cuidado con CascadeType.REMOVE: Si un Cliente tiene mascotas y esas mascotas tienen turnos, eliminar el cliente en cascada intenta borrar las mascotas, pero fallará si los turnos tienen FK restrictiva hacia mascota. Preferir siempre baja lógica (activo = false) en sistemas con datos relacionados.

@OneToOne
DIAPOSITIVA 17 — Relación uno a uno
📋 ¿Cuándo usar @OneToOne?

Cuando exactamente un registro de una tabla se corresponde con exactamente un registro de otra. Ejemplo: un Usuario tiene un Perfil, un Pedido tiene una Factura, un Empleado tiene un LegajoMedico.

📄 Usuario.java — lado dueño (tiene FK)
@Entity
public class Usuario {
  @Id @GeneratedValue(...)
  private Long id;
  private String username;

  // El lado dueño tiene la FK
  @OneToOne(
    cascade  = CascadeType.ALL,
    fetch    = FetchType.LAZY,
    optional = false
  )
  @JoinColumn(name = "perfil_id", unique = true)
  private Perfil perfil;
  // tabla usuario → columna perfil_id (FK + UNIQUE)
}
📄 Perfil.java — lado inverso
@Entity
public class Perfil {
  @Id @GeneratedValue(...)
  private Long id;
  private String bio;
  private String foto;

  // El lado inverso usa mappedBy
  @OneToOne(mappedBy = "perfil")
  private Usuario usuario;
  // tabla perfil → NO tiene ninguna FK extra
}

@ManyToMany y tabla intermedia
DIAPOSITIVA 18 — Relación muchos a muchos
📋 Ejemplo: Veterinario ↔ Especialidad

Un veterinario puede tener múltiples especialidades y una especialidad puede ser practicada por múltiples veterinarios. JPA crea automáticamente la tabla intermedia veterinario_especialidad.

📄 Veterinario.java — lado dueño
@Entity
public class Veterinario {
  @Id @GeneratedValue(...)
  private Long id;
  private String nombre;

  @ManyToMany(fetch = FetchType.LAZY)
  @JoinTable(
    name               = "veterinario_especialidad",
    joinColumns        = @JoinColumn(name = "vet_id"),
    inverseJoinColumns = @JoinColumn(name = "esp_id")
  )
  private List<Especialidad> especialidades
    = new ArrayList<>();
  // tabla: veterinario_especialidad(vet_id, esp_id)
}
📄 Especialidad.java — lado inverso
@Entity
public class Especialidad {
  @Id @GeneratedValue(...)
  private Long id;
  private String nombre;

  @ManyToMany(
    mappedBy = "especialidades" // campo en Veterinario
  )
  private List<Veterinario> veterinarios
    = new ArrayList<>();
  // NO usa @JoinTable — lo define el lado dueño
}
💡
Tabla intermedia con atributos: Si la tabla intermedia necesita campos propios (ej: fecha_desde), se crea una entidad intermedia (VeterinarioEspecialidad) con sus propias relaciones @ManyToOne a cada lado. Esto se llama entidad de unión.

JPQL con relaciones: path expressions
DIAPOSITIVA 19 — Navegación por relaciones en JPQL
💡
Path expression: En JPQL se puede navegar a través de relaciones usando el punto, igual que en Java: m.cliente.apellido. JPA genera automáticamente el JOIN SQL necesario.
Consultas JPQL con navegación de relaciones
// 1. Mascotas de un cliente específico (por ID)
em.createQuery(
  "SELECT m FROM Mascota m WHERE m.cliente.id = :clienteId",
  Mascota.class)
  .setParameter("clienteId", 1L)
  .getResultList();
// → SELECT * FROM mascota WHERE cliente_id = 1

// 2. Mascotas de clientes activos (navegar 2 niveles)
em.createQuery(
  "SELECT m FROM Mascota m WHERE m.cliente.activo = true",
  Mascota.class)
  .getResultList();

// 3. Turnos de una mascota, con JOIN FETCH del cliente del dueño
em.createQuery(
  "SELECT t FROM Turno t "
+ "JOIN FETCH t.mascota m "
+ "JOIN FETCH m.cliente "
+ "WHERE t.mascota.id = :mascotaId "
+ "ORDER BY t.fechaHora DESC",
  Turno.class)
  .setParameter("mascotaId", mascotaId)
  .getResultList();

// 4. Clientes que tienen mascotas activas
em.createQuery(
  "SELECT DISTINCT c FROM Cliente c "
+ "JOIN c.mascotas m WHERE m.activo = true",
  Cliente.class)
  .getResultList();

// 5. Cantidad de mascotas por cliente
em.createQuery(
  "SELECT c.apellido, COUNT(m) FROM Cliente c "
+ "LEFT JOIN c.mascotas m GROUP BY c.id ORDER BY c.apellido")
  .getResultList();
DIAPOSITIVA 20 — Tabla: tipos de JOIN en JPQL
TipoJPQLResultadoCuándo usar

DIAPOSITIVA 21 — El problema de mostrar relaciones en selectores
🖥️ Patrón: selector con ID + resolución en el Servicio

En un formulario JSF, cuando hay un <h:selectOneMenu> para elegir el cliente de una mascota, el selector retorna un ID (Long). El Bean guarda ese ID y el Servicio resuelve la relación buscando el objeto completo con em.find().

📄 mascotas/abm.xhtml — Selector de Cliente
<!-- El selector devuelve el ID del cliente -->
<h:selectOneMenu
  id="cliente"
  value="#{mascotaBean.clienteIdSeleccionado}"
  required="true">

  <f:selectItem
    itemValue=""
    itemLabel="— Seleccione un cliente —"
    noSelectionOption="true"/>

  <f:selectItems
    value="#{mascotaBean.clientes}"
    var="cli"
    itemValue="#{cli.id}"           ← ID (Long)
    itemLabel="#{cli.apellido}, #{cli.nombre}"/>

  <f:converter converterId="jakarta.faces.Long"/>
  <!-- Converter necesario: String del HTML → Long de Java -->
</h:selectOneMenu>
📄 MascotaBean.java + MascotaServicio.java
// En el Bean: guarda el ID seleccionado
private Long clienteIdSeleccionado; // ← solo el ID

public void guardar() {
  // Le pasa el ID al servicio
  mascotaServicio.guardar(
      mascotaSeleccionada,
      clienteIdSeleccionado // ← ID del selector
  );
}

// En el Servicio: resuelve la relación
@Transactional
public void guardar(Mascota mascota, Long clienteId) {
  // Busca el objeto Cliente completo en la BD
  Cliente cliente = clienteDAO.buscarPorId(clienteId);
  if (cliente == null || !cliente.isActivo())
    throw new IllegalArgumentException("Cliente inválido");

  mascota.setCliente(cliente); // ← setea el objeto, no el ID

  if (mascota.getId() == null) mascotaDAO.guardar(mascota);
  else                         mascotaDAO.actualizar(mascota);
}

<!-- IMPORTANTE: para que t.mascota.cliente funcione sin N+1
     el DAO debe usar JOIN FETCH en la NamedQuery -->
⚠️
Regla crítica: Si en la vista se navega t.mascota.cliente.apellido, la consulta del DAO debe tener JOIN FETCH t.mascota JOIN FETCH m.cliente. Sin eso, cada acceso a cliente genera una consulta extra → problema N+1.

DIAPOSITIVA 23 — Cuadro de decisión: ¿qué anotación usar?
🤔 ¿Qué anotación necesito?
¿Cuántos A tienen cuántos B?

1 A → 1 B                 → @OneToOne   (en el lado con FK)
1 A → muchos B            → @OneToMany  (en A, mappedBy)  
Muchos A → 1 B            → @ManyToOne  (en A, tiene FK)  ← más común
Muchos A → muchos B       → @ManyToMany (en uno, @JoinTable)

¿Quién tiene la clave foránea en la tabla?
  El lado "Muchos" → @ManyToOne

¿Cuándo poner @OneToMany?
  En el lado "Uno", con mappedBy apuntando al campo @ManyToOne del lado "Muchos"

DIAPOSITIVA 25 — Errores frecuentes con relaciones JPA
ErrorCausaSolución
LazyInitializationException Acceder a colección LAZY fuera del contexto JPA (fuera de la transacción) Usar JOIN FETCH en la consulta DAO, o cargar la relación dentro de la transacción
ConverterException en selectOneMenu El itemValue="#{cli.id}" viene como String y el Bean espera Long Agregar <f:converter converterId="jakarta.faces.Long"/>
ConstraintViolationException Intentar guardar Mascota sin setear el Cliente (FK null) Llamar mascota.setCliente(cliente) antes de em.persist()
StackOverflowError toString()/hashCode() con referencia circular A→B→A No incluir la relación inversa en toString(). Usar @EqualsAndHashCode(exclude=...)
N+1 queries sin error visible LAZY + acceso en bucle desde la vista sin JOIN FETCH Activar eclipselink.logging.level.sql=FINE para ver las queries y agregar JOIN FETCH
mappedBy incorrecto mappedBy="clientes" pero el campo en la otra clase se llama "cliente" mappedBy debe coincidir exactamente con el nombre del campo Java (case sensitive)

En TurnoDAO, asegurarse de que todas las @NamedQuery que devuelven listas tengan JOIN FETCH t.mascota JOIN FETCH m.cliente. Volver a contar las queries.

Listar mascotas de un cliente

En MascotaDAO, crear el método listarPorCliente(Long clienteId) usando la @NamedQuery con WHERE m.cliente.id = :clienteId. Probarlo desde un endpoint de prueba.

En turnos/abm.xhtml, el selector debe mostrar el nombre de la mascota y el apellido de su dueño: itemLabel="#{m.nombre} — #{m.cliente.apellido}". Verificar que no genera N+1 usando la NamedQuery Mascota.findActivasConCliente con JOIN FETCH.
